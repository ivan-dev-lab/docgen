# Docgen

`Docgen` — CLI-инструмент для автоматического анализа кодовой базы и генерации документации с двухслойной архитектурой: сначала собираются проверяемые факты о проекте, затем на их основе строится обычная Markdown-документация и углубленные LLM-объяснения.

Проект решает практическую задачу: быстро получить понятное описание чужой или собственной кодовой базы без ручного разбора каждого файла. При этом `Docgen` не отдает всю работу языковой модели напрямую. Сначала он строит фактический слой: файлы, функции, классы, зависимости, роли модулей, покрытие анализа, ограничения и навигационные документы. Только после этого LLM получает подготовленный контекст и генерирует объяснения с учетом бюджета, проверки фактической опоры и семантической валидации.

Главная идея проекта:

```text
кодовая база
  -> детерминированный анализ
  -> фактические JSON-артефакты
  -> Markdown-документация
  -> план объяснения
  -> LLM-объяснение модулей
  -> проверка качества и фактической опоры
  -> пакетная генерация документации
````

Такой подход снижает риск “красивой, но выдуманной” документации: LLM используется не как источник фактов, а как слой объяснения поверх уже собранного контекста.

---

## Что умеет проект

`Docgen` поддерживает полный цикл подготовки документации:

* анализирует структуру проекта;
* строит инвентарь файлов;
* находит функции, классы, методы и их сигнатуры;
* извлекает зависимости между файлами и внешними пакетами;
* выделяет кандидаты на логические модули;
* оценивает покрытие анализа и ограничения;
* сохраняет машинно-читаемые JSON-артефакты;
* генерирует фактическую Markdown-документацию;
* строит навигационные документы по файлам, модулям и ролям частей проекта;
* формирует `explain-plan` для LLM-слоя;
* подключается к OpenRouter как LLM-провайдеру;
* выполняет smoke-проверку провайдера;
* генерирует углубленное объяснение отдельного модуля;
* контролирует бюджет входного контекста до вызова модели;
* умеет сокращать контекст по прозрачному плану;
* валидирует сгенерированный текст по обязательным секциям, осторожности формулировок и фактической опоре;
* использует fail-safe верификацию, чтобы не выдавать непроверенный результат как надежный;
* поддерживает пакетную генерацию документации по нескольким модулям;
* использует cache для повторных запусков и экономии LLM-вызовов;
* поддерживает пакетную проверку сгенерированных объяснений.

## Для чего это нужно

Документация по проектам часто сталкивается с несколькими проблемами:

1. **Она устаревает.**
   Код меняется быстрее, чем документация.

2. **Она неполная.**
   README обычно описывает только верхний уровень, а внутренняя структура проекта остается непонятной.

3. **Она плохо подходит для входа в проект.**
   Новому разработчику нужно понять, где точка входа, какие модули за что отвечают, где тесты, какие файлы связаны между собой.

4. **LLM может уверенно выдумывать.**
   Если просто дать модели проект и попросить “объясни”, она может написать правдоподобный, но неточный текст.

`Docgen` решает это через разделение ответственности:

* факты извлекаются детерминированно;
* документация рендерится из артефактов;
* LLM получает уже подготовленный контекст;
* сгенерированный текст дополнительно проверяется.

## Основной pipeline

Типовой сценарий работы:

```text
1. analyze
   исходный проект -> JSON-артефакты анализа

2. render
   JSON-артефакты -> фактическая Markdown-документация

3. explain-plan
   analysis + generated docs -> структурированный план для LLM

4. llm-smoke
   проверка подключения к OpenRouter

5. explain-module
   генерация углубленного объяснения одного модуля

6. verify-module
   семантическая и фактическая проверка результата

7. explain-batch
   пакетная генерация объяснений по нескольким модулям с cache

8. verify-batch
   пакетная проверка сгенерированных объяснений
```

## Команда `analyze`

Команда `analyze` выполняет детерминированный анализ кодовой базы и сохраняет набор JSON-артефактов.

Пример:

```bash
docgen analyze . --output .docgen-analysis
```

На выходе формируются:

```text
inventory.json
function-index.json
dependency-graph.json
module-candidates.json
analysis-summary.json
coverage-report.json
artifact-manifest.json
```

### Что содержит `inventory.json`

Инвентарь файлов проекта:

* путь к файлу;
* расширение;
* тип файла;
* язык;
* размер;
* признак тестового файла;
* признак конфигурационного файла;
* возможная точка входа;
* глубина анализа;
* признак fixture/sample/generated файла.

### Что содержит `function-index.json`

Индекс найденных сущностей:

* функции;
* классы;
* методы;
* сигнатуры;
* параметры;
* аннотации типов;
* docstring;
* строки начала и конца;
* контекст класса или модуля;
* признак экспортируемости, если его можно определить.

### Что содержит `dependency-graph.json`

Граф зависимостей:

* внутренние импорты;
* внешние зависимости;
* стандартная библиотека;
* unresolved-зависимости;
* связи между файлами.

Для Python используется AST-анализ. Для JavaScript и TypeScript применяется более осторожный синтаксический анализ импортов и экспортируемых сущностей.

### Что содержит `module-candidates.json`

Кандидаты на логические модули проекта. Этот артефакт нужен, чтобы документация строилась не только по отдельным файлам, но и по смысловым зонам проекта.

Например:

```text
CLI layer
analysis layer
render layer
LLM layer
tests
configuration
```

### Что содержит `coverage-report.json`

Отчет о покрытии анализа:

* какие файлы проанализированы глубоко;
* какие только частично;
* какие обработаны поверхностно;
* какие ограничения обнаружены;
* какие участки проекта не стоит описывать слишком уверенно.

Это важная часть проекта: инструмент явно фиксирует границы своего знания и не делает вид, что понял то, чего не смог извлечь.

## Фильтрация и cleanup

`Docgen` не должен тащить в анализ все подряд. Поэтому в анализаторе есть слой исключений и классификации файлов.

Обычно исключаются:

```text
.git/
.venv/
venv/
node_modules/
dist/
build/
__pycache__/
.pytest_cache/
.mypy_cache/
.ruff_cache/
coverage/
htmlcov/
.docgen-analysis/
```

Также отдельно обрабатываются:

* тесты;
* fixtures;
* sample-проекты;
* generated outputs;
* packaging metadata;
* конфигурационные файлы;
* документация;
* ассеты.

Для этого в CLI есть флаги:

```bash
docgen analyze . --include-tests true
docgen analyze . --include-fixtures false
docgen analyze . --include-generated false
```

Такой подход нужен, чтобы документация строилась по реальному исходному коду, а не по временным артефактам, результатам прошлых запусков или служебным файлам окружения.

## Команда `render`

Команда `render` превращает JSON-артефакты анализа в Markdown-документацию.

Пример:

```bash
docgen render --analysis .docgen-analysis --output docs/generated
```

Рендеринг отделен от анализа. Это дает несколько преимуществ:

* анализ можно выполнить один раз;
* Markdown-документацию можно перегенерировать отдельно;
* JSON-артефакты остаются машинно-читаемым источником правды;
* человекочитаемые документы не смешиваются с внутренними данными анализа.

Рендер формирует документацию по проекту, файлам, кандидатам на модули, покрытию анализа и связям между частями кодовой базы.

---

## Навигационная документация

После этапа factual render проект формирует не только общий отчет, но и навигационные документы.

Цель навигационного слоя — помочь быстро понять проект:

* какие файлы есть;
* какие из них являются точками входа;
* какие относятся к анализу;
* какие относятся к рендеру;
* где находится CLI;
* где находится LLM-слой;
* какие файлы похожи на тесты;
* какие модули играют ключевую роль.

Это особенно полезно для крупных проектов, где обычный README не помогает быстро войти в структуру.

---

## Команда `explain-plan`

Команда `explain-plan` строит структурированный план для последующей LLM-генерации.

Пример:

```bash
docgen explain-plan --analysis .docgen-analysis --docs docs/generated --output docs/generated/explain-plan.json
```

`explain-plan` не генерирует финальный объясняющий текст. Его задача — подготовить контекст:

* какие модули можно объяснять;
* какие документы уже есть;
* какие файлы связаны с модулем;
* какие ограничения анализа нужно учитывать;
* какие factual-документы можно использовать как основу;
* какие части проекта стоит объяснять в первую очередь.

Этот этап нужен, чтобы LLM не работала с сырым хаосом файлов, а получала заранее организованный контекст.

---

## Согласованность analyze / render / explain-plan

В проекте отдельно усилена согласованность между тремя слоями:

```text
analyze -> render -> explain-plan
```

Это означает:

* `analyze` сохраняет стабильные артефакты;
* `render` публикует документацию с машинно-читаемыми связями;
* `explain-plan` использует эти связи, а не пытается угадывать пути к файлам;
* если нужная связь отсутствует, система должна явно предупредить или использовать контролируемый fallback.

Такой подход снижает риск рассинхронизации: например, когда анализатор знает один путь к файлу, рендерер публикует другой, а LLM-план потом не может надежно сопоставить модуль с документацией.

---

## OpenRouter provider и `llm-smoke`

LLM-слой подключается через OpenRouter.

Smoke-проверка нужна, чтобы проверить провайдера отдельно от генерации документации.

Сухой запуск:

```bash
docgen llm-smoke --provider openrouter --dry-run
```

Реальный smoke-запрос:

```bash
docgen llm-smoke --provider openrouter --model openrouter/provider/model --max-tokens 64
```

Команда возвращает JSON-ответ:

* провайдер;
* модель;
* статус;
* preview ответа;
* usage;
* наличие reasoning-полей;
* наличие reasoning details, если провайдер их возвращает.

Smoke-слой специально не генерирует документацию. Его задача — проверить, что LLM-провайдер доступен и конфигурация работает.

---

## Команда `explain-module`

`explain-module` генерирует углубленное объяснение одного модуля на основе `explain-plan` и фактических документов.

Пример:

```bash
docgen explain-module \
  --plan docs/generated/explain-plan.json \
  --module llm \
  --output docs/enhanced/modules/module-llm.md
```

Задача команды — не просто пересказать файлы, а объяснить роль модуля:

* за что отвечает модуль;
* какие файлы в него входят;
* как он связан с остальной системой;
* какие входы и выходы у слоя;
* какие ограничения есть у анализа;
* что можно утверждать уверенно, а что требует осторожности;
* какие части поведения подтверждаются factual-артефактами.

Результат сохраняется как enhanced Markdown-документ. Дополнительно сохраняется metadata по запуску: модель, провайдер, бюджет, статусы валидации и служебная информация.

## Budget preflight

Перед LLM-вызовом `Docgen` оценивает размер полного prompt payload.

Это нужно, чтобы не отправлять запрос, который:

* превышает лимит модели;
* может быть обрезан;
* приведет к дорогому или нестабильному запуску;
* ухудшит качество ответа из-за перегруженного контекста.

Система учитывает:

* исходный `explain-plan`;
* связанные factual-документы;
* выбранный модуль;
* инструкции для модели;
* safety margin;
* оценку входных токенов;
* ожидаемый размер ответа.

Если бюджет превышен, live-вызов блокируется без явного разрешения.

---

## Transparent reduction plan

Если контекст слишком большой, система не должна просто “обрезать последние файлы”. Вместо этого применяется прозрачный план сокращения.

Reduction plan должен показывать:

* что было включено изначально;
* какие части были сокращены;
* почему они были сокращены;
* какие документы остались в prompt;
* какие риски появились после сокращения контекста.

Это важно для доверия к LLM-ответу: пользователь должен понимать, на каком основании было сгенерировано объяснение и какой контекст до модели реально дошел.

---

## Semantic validation

После генерации текста выполняется семантическая проверка результата.

Проверяются:

* наличие обязательных секций;
* отсутствие чрезмерно уверенных утверждений;
* наличие осторожных формулировок там, где анализ неполный;
* связь объяснения с factual-артефактами;
* отсутствие явных неподтвержденных заявлений;
* достаточная полезность текста для понимания модуля.

Идея простая: LLM-ответ не считается хорошим только потому, что он красиво написан. Он должен быть структурированным, полезным и не выходить за пределы подтвержденного контекста.

---

## Verifier fail-safe

В проекте реализован fail-safe подход к проверке.

Если верификатор не может надежно подтвердить качество результата, система не должна маркировать результат как полностью надежный.

Возможные состояния:

* проверка пройдена;
* проверка пройдена с предупреждениями;
* проверка не пройдена;
* проверка не смогла надежно выполниться.

Fail-safe нужен для честного поведения: лучше явно показать, что результат требует проверки, чем уверенно выдать слабое объяснение как готовую документацию.

---

## Single-module semantic verifier

Для одного модуля работает отдельный верификатор, который сравнивает generated explanation с доступным контекстом.

Он проверяет:

* соответствует ли текст выбранному модулю;
* не подменяет ли он один слой проекта другим;
* не заявляет ли он неподтвержденные возможности;
* не игнорирует ли ограничения анализа;
* есть ли в тексте практическая польза для читателя;
* достаточно ли ясно объяснена роль модуля.

Это особенно важно для проектов, где LLM может легко перепутать намерение архитектуры и фактическое состояние кода.

---

## Batch generation with cache

После single-module generation проект поддерживает пакетную генерацию.

Идея batch-режима:

```text
explain-plan
  -> список модулей
  -> генерация enhanced docs
  -> metadata по каждому модулю
  -> cache
  -> пропуск неизменившихся результатов при повторном запуске
```

Cache нужен для нескольких целей:

* не платить повторно за одинаковые LLM-вызовы;
* ускорять повторную генерацию;
* сохранять стабильность результата;
* запускать документационный pipeline итеративно;
* отдельно обновлять только изменившиеся модули.

Пакетная генерация полезна, когда нужно получить не одно объяснение, а набор enhanced-документов по всему проекту.

---

## Batch verification

Пакетная проверка применяет верификацию не к одному модулю, а к набору generated-документов.

Она позволяет увидеть:

* какие модули прошли проверку;
* где есть предупреждения;
* какие результаты требуют ручного просмотра;
* какие документы стоит перегенерировать;
* где контекста было недостаточно;
* где сработал fail-safe.

Результаты batch verification можно использовать как quality gate перед публикацией документации.

## Структура проекта

Упрощенная структура:

```text
src/docgen/
  analyzer.py
  renderer.py
  cli.py

  llm/
    config.py
    schemas.py
    prompts.py
    context_builder.py
    explain_plan.py
    openrouter_provider.py
    verifier.py
    reduction.py
    cache.py
    batch.py

tests/
  test_analyzer_*.py
  test_renderer_*.py
  test_explain_plan_*.py
  test_llm_provider_*.py
  test_explain_module_*.py
  test_semantic_verifier_*.py
  test_batch_*.py
```

Основные зоны ответственности:

* `analyzer.py` — статический анализ проекта и сбор JSON-артефактов;
* `renderer.py` — генерация factual Markdown-документации;
* `cli.py` — командный интерфейс;
* `llm/config.py` — конфигурация LLM-провайдера;
* `llm/openrouter_provider.py` — интеграция с OpenRouter;
* `llm/explain_plan.py` — подготовка плана объяснения;
* `llm/context_builder.py` — сбор контекста для LLM;
* `llm/prompts.py` — системные и пользовательские инструкции;
* `llm/reduction.py` — сокращение контекста при превышении бюджета;
* `llm/verifier.py` — семантическая проверка результата;
* `llm/cache.py` — cache для повторных LLM-запусков;
* `llm/batch.py` — пакетная генерация и проверка.

## Базовый сценарий использования

```bash
docgen analyze . --output .docgen-analysis

docgen render \
  --analysis .docgen-analysis \
  --output docs/generated

docgen explain-plan \
  --analysis .docgen-analysis \
  --docs docs/generated \
  --output docs/generated/explain-plan.json

docgen llm-smoke \
  --provider openrouter \
  --dry-run

docgen explain-module \
  --plan docs/generated/explain-plan.json \
  --module llm \
  --output docs/enhanced/modules/module-llm.md

docgen verify-module \
  --plan docs/generated/explain-plan.json \
  --module llm \
  --document docs/enhanced/modules/module-llm.md

docgen explain-batch \
  --plan docs/generated/explain-plan.json \
  --output docs/enhanced/modules \
  --cache docs/enhanced/cache

docgen verify-batch \
  --plan docs/generated/explain-plan.json \
  --docs docs/enhanced/modules \
  --output docs/enhanced/verification
```

## Конфигурация LLM

Для OpenRouter используется переменная окружения:

```bash
OPENROUTER_API_KEY=...
```

или другой поддерживаемый проектом вариант окружения, если он задан в конфигурационном слое.

Smoke-проверка позволяет проверить конфигурацию без генерации документации:

```bash
docgen llm-smoke --provider openrouter --dry-run
```

Для реального запроса можно указать модель:

```bash
docgen llm-smoke \
  --provider openrouter \
  --model openrouter/provider/model \
  --max-tokens 64
```

## Метаданные LLM-запусков

LLM-команды сохраняют не только итоговый Markdown, но и служебную информацию.

Обычно metadata включает:

* провайдера;
* модель;
* параметры запуска;
* оценку входного бюджета;
* использованный reduction plan;
* статус semantic validation;
* наличие предупреждений;
* сведения о cache;
* usage, если провайдер его вернул;
* статус verifier;
* путь к исходному explain-plan;
* путь к итоговому документу.

Это нужно, чтобы результат можно было проверить и воспроизвести.

---

## Почему LLM-слой отделен от factual-слоя

В проекте принципиально не смешиваются два типа данных:

1. **Факты, извлеченные из кода.**
   Это JSON-артефакты, Markdown-рендер и навигационные документы.

2. **Интерпретация и объяснение.**
   Это LLM-generated enhanced documentation.

Такое разделение важно по нескольким причинам:

* факты можно проверить;
* LLM-ответ можно ограничить контекстом;
* можно явно показать, где модель опирается на анализ, а где делает осторожную интерпретацию;
* можно перегенерировать объяснение без повторного анализа проекта;
* можно обновлять только изменившиеся модули;
* можно использовать verification как отдельный quality gate.

## Ограничения

Даже с LLM-слоем проект не должен восприниматься как инструмент, который “понимает код лучше разработчика”.

Ограничения:

* статический анализ не всегда понимает динамический код;
* JavaScript/TypeScript анализ проще, чем Python AST-анализ;
* качество модульных кандидатов зависит от структуры проекта;
* LLM-объяснения требуют проверки, особенно на сложной бизнес-логике;
* сокращение контекста может ухудшить полноту объяснения;
* verifier снижает риск ошибок, но не заменяет человеческое ревью;
* batch generation повышает масштабируемость, но требует контроля качества.

---

## Ключевой принцип

`Docgen` строится вокруг идеи:

> LLM должна объяснять проект на основе проверяемого контекста, а не выдумывать документацию с нуля.

Поэтому проект сначала создает factual foundation, затем строит документацию, затем формирует explain-plan, затем подключает LLM и только после этого проверяет качество результата.

Это делает систему более надежной, чем простой подход “отправить код модели и попросить написать README”.
